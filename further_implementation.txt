CINELINGUS DEEP TRANSFORMATION PROGRAM
OVERALL IMPLEMENTATION PLAN
===========================

## Purpose

This document defines the phased development program required to evolve
Cinelingus from a reflective dialogue-transformation engine into a system that
can represent, alter, verify, and repair deeper cinematic properties.

This is not a single implementation work order.

It is the governing program plan from which a separate, detailed work order
will be written for each phase.

Each phase must:

1. Begin from the verified state produced by the previous phase.
2. Deliver one coherent capability or vertical slice.
3. Include measurable acceptance criteria.
4. Preserve existing working behavior.
5. Produce evidence sufficient to shape the following phase.
6. Avoid implementing later abstractions prematurely.
7. End with a builder capability report and recommendation.

The program should remain adaptable.

Later phases may be revised, divided, reordered, or cancelled based on measured
results from earlier phases.

======================================================================
CURRENT VERIFIED BASELINE
=========================

Cinelingus currently provides:

* Whisper transcription
* Pyannote speaker diarization
* shot and transition analysis
* cinematic moments
* cross-modal Performance objects
* performance-aware Translation scheduling
* full-timeline audio and video rendering
* rendered-output retranscription
* residue verification
* placement-level quality decisions
* failure taxonomy
* failure-specific repair strategies
* multi-pass editorial refinement
* rollback of measured regressions
* editorial memory
* repair telemetry and preflight elimination
* deterministic corpus inventory and evaluation plans
* 64 compatible source films
* approximately 44.34 hours of corpus media
* read-only corpus protection
* 498 passing tests at the latest reported milestone

Latest measured repair result:

* initial quality: 0.6732
* final quality: 0.6880
* accepted repairs: 5
* candidate survival: 53.1%
* unresolved placements: 185
* pre-render eliminations: 14
* safely restored regressions: 15

The immediate technical limitation is no longer the absence of a reflective
pipeline.

The limitations are:

* repair effectiveness remains uneven
* many placements remain unresolved
* coordinated repair is incomplete
* visible speakers are not reliably identified
* semantic meaning is represented weakly
* cinematic functions are not explicit
* no unified Cinematic Model exists
* transformation laws cannot yet target character, meaning, relationship, or
  narrative structure directly

======================================================================
PROGRAM PRINCIPLES
==================

1. VERTICAL SLICES BEFORE BROAD FRAMEWORKS

Every phase should prove one end-to-end capability:

observe
represent
transform
render
verify
repair

Do not build large abstract systems without a transformation that demonstrates
their value.

2. OBSERVATION BEFORE INTERPRETATION

Represent measurable evidence before assigning cinematic meaning.

Examples:

* establish recurring visual tracks before claiming character identity
* classify turn behavior before claiming relationship meaning
* measure interaction change before claiming friendship or hostility

3. UNCERTAINTY IS A VALID RESULT

Every inferred property must support:

* confidence
* evidence
* contradiction
* unknown
* abstention

A confident wrong answer is more damaging than an explicit unknown.

4. ADDITIVE ARCHITECTURE

New capabilities should enrich the current pipeline.

Do not rewrite the working transcription, scheduling, rendering, verification,
or editorial systems unless evidence proves that replacement is necessary.

5. NO CAPABILITY THEATER

The system must disclose whether an apparatus is operating with:

* full capability
* degraded capability
* legacy heuristics
* unavailable capability

Do not present heuristic behavior as semantic or cinematic understanding.

6. MEASURABLE TRANSFORMATION LAWS

A transformation is not complete merely because it produces an output.

The system must measure whether the intended property changed and whether
preserved properties remained intact.

7. CORPUS-GROUNDED DEVELOPMENT

The local movie corpus should be used as:

* a development laboratory
* a calibration set
* a regression corpus
* a source of difficult cases

It must not become an uncontrolled exhaustive workload.

8. SOURCE MEDIA REMAINS READ-ONLY

No phase may write into, rename, move, delete, or alter source media.

## Goal

Complete and calibrate the repair system before adding deeper cinematic
constraints.

## Primary Deliverables

* evidence-selected repair excerpts
* benchmark cases for every repair strategy
* coordinated neighborhood repair
* performance-group repair
* shared-boundary and shared-ambience repair
* candidate-level interruption recovery
* multi-pair corpus calibration
* quality-threshold calibration
* revised run-level acceptance contract
* repair strategy effectiveness report

## Required Capabilities

The engine must be able to repair a bounded group of related placements when
an atomic change cannot resolve:

* duration pressure
* turn continuity
* speaker continuity
* transition artifacts
* masking
* residual dialogue
* shared room tone
* performance-level mismatch

## Evaluation Scope

At minimum:

* animation to animation
* live action to live action
* animation to live action
* dialogue-dense material
* sparse dialogue
* music-heavy dialogue
* action-heavy visuals
* rapid turn-taking
* short-form material
* feature-length material

## Exit Gate

Proceed only when:

* every repair strategy has an executable benchmark case
* neighborhood repair is working
* performance-group repair is working
* results are measured across multiple corpus strata
* final placement states accurately describe outcomes
* hard failures cannot be concealed by an average quality score
* repair work can resume at candidate or placement granularity
* known successful repairs are covered by regression tests
* at least two materially different film pairs show measured benefit

## Required End-of-Phase Report

The builder must report:

* final implementation status
* repair effectiveness by strategy
* successful and failed neighborhood repairs
* unresolved failure distribution
* quality calibration findings
* corpus strata tested
* regressions discovered
* recommended scope for Phase 1

PHASE 1 — MINIMAL CINEMATIC MODEL CORE
======================================

## Goal

Create a stable, versioned representation that unifies existing Cinelingus
artifacts without yet attempting semantic scene or character understanding.

This phase is architectural, but deliberately narrow.

## Primary Deliverables

A minimal FilmModel containing references to:

* media identity
* timeline
* shots
* transitions
* speech passages
* speaker clusters
* dialogue turns
* performances
* cinematic moments
* editorial observations
* provenance
* confidence

Additional deliverables:

* stable film-local entity IDs
* versioned schema
* deterministic serialization
* cache integration
* model validation
* migration from existing artifacts
* human-readable model report
* capability declarations

## Explicitly Excluded

Do not add during this phase:

* semantic embeddings
* character recognition
* active-speaker detection
* relationship inference
* dialogue-function classification
* scene meaning
* new public transformations

## Proof Requirement

Use the model to reproduce one existing Translation schedule and its provenance
without changing the rendered result.

The purpose is to prove that the model can become the common representation
without destabilizing current execution.

## Exit Gate

Proceed only when:

* current artifacts can be represented without loss
* IDs remain stable under unchanged analysis signatures
* cache invalidation works correctly
* legacy filters continue to run
* existing schedules can be traced through the model
* the model adds no fabricated semantic claims
* all model fields have provenance
* serialization and reload are deterministic

## Required End-of-Phase Report

The builder must report:

* final schema
* fields intentionally omitted
* migration risks
* cache impact

* compatibility findings

PHASE 3 — DIALOGUE FUNCTION AND FUNCTION-PRESERVING TRANSLATION
===============================================================

## Goal

Represent what a dialogue turn appears to do within a conversational exchange,
then prove that this property can be preserved through transformation.

## Initial Taxonomy

The first taxonomy should remain compact:

* question
* answer
* acknowledgment
* agreement
* disagreement
* refusal
* request
* command
* warning
* accusation
* defense
* explanation
* reassurance
* confession
* threat
* revelation
* interruption
* deflection
* narration
* unknown

Taxonomy changes should be based on corpus evidence rather than theoretical
completeness.

## Primary Deliverables

* versioned dialogue-function taxonomy
* probabilistic function inference
* context-aware classification
* confidence and abstention
* rule/model provenance
* reviewed calibration set
* function compatibility scoring
* dialogue-function verification
* dialogue-function repair behavior

## Proof Transformation

Function-Preserving Translation

The system should attempt to preserve the destination function sequence while
replacing dialogue content.

Examples:

* questions remain questions
* answers remain answers
* refusals remain refusals
* interruptions remain interruptions
* explanations remain explanations

## Required Verification

For every transformed turn:

* original function probabilities
* replacement function probabilities
* preservation score
* confidence
* transcript completeness
* performance compatibility
* rendered verification

## Required Repair

When a function mismatch occurs:

* search another donor turn
* search another turn sequence when context determines function
* retain the best-known version when confidence is insufficient
* never repair semantic function by violating hard timing or completeness gates

## Exit Gate

Proceed only when:

* the taxonomy is calibrated on reviewed examples
* unknown and ambiguous cases are preserved
* Function-Preserving Translation runs end to end
* function preservation is measurably better than standard Translation
* rendered quality remains within agreed tolerance
* function-specific repair succeeds in benchmark cases
* the architecture does not require natural-language directing or an LLM

## Required End-of-Phase Report

The builder must report:

* confusion matrix or equivalent calibration evidence
* taxonomy weaknesses
* function-preservation score
* effect on technical quality
* successful and failed function repairs
* reviewer findings
* recommendation for Phase 4 visual work

PHASE 4 — VISUAL TRACKS AND ACTIVE-SPEAKER EVIDENCE
===================================================

## Goal

Move from isolated face and mouth observations to persistent visual tracks and
time-local active-speaker evidence.

This phase does not yet promise reliable film-wide character identity.

## Primary Deliverables

* persistent within-shot visual tracks
* conservative across-shot track linking
* improved mouth-motion representation
* head-motion compensation
* per-track speaking likelihood
* off-screen-speech probability
* listening-state probability
* multiple-face handling
* explicit unknown state
* active-speaker evaluation set
* live-action and animation comparison

## Implementation Sequence

Stage 1:

* strengthen existing face and mouth evidence
* construct stable within-shot tracks
* evaluate track fragmentation and identity switches

Stage 2:

* add conservative cross-shot continuity
* use temporal and shot-pattern evidence
* avoid forced merging

Stage 3:

* evaluate a local pretrained active-speaker model if it materially improves
  results over the strengthened heuristic baseline

Do not commit to a new model before the baseline comparison is complete.

## Proof Requirement

Create a report identifying, for selected excerpts:

* visible tracks
* likely active speaker
* likely listeners
* likely off-screen speech
* uncertain intervals
* supporting evidence

Do not yet use this evidence as a hard scheduling constraint.

## Exit Gate

Proceed only when:

* within-shot tracking is stable enough for evaluation
* false confident attribution is measured
* animation and live-action results are separated
* the system abstains when evidence is weak
* active-speaker evidence improves on the prior mouth-motion heuristic
* model or heuristic provenance is recorded
* no real-world identity recognition is used

## Required End-of-Phase Report

The builder must report:

* tracking method
* active-speaker method
* precision and recall
* abstention rate
* false confident attribution rate
* live-action versus animation performance
* runtime cost
* recommendation for Phase 5 character continuity

PROGRAM-WIDE QUALITY GATES
==========================

Every phase-specific work order must include:

1. PRECISE BASELINE

Document the repository state, current metrics, relevant modules, and known
limitations.

2. LIMITED SCOPE

State what the phase will and will not implement.

3. TECHNICAL DECISIONS

Select concrete models, schemas, algorithms, thresholds, and interfaces where
required.

4. VERTICAL PROOF

Deliver one complete capability from analysis through verified output.

5. CORPUS EVALUATION

Test representative material rather than one convenient example.

6. REGRESSION PROTECTION

Preserve existing tests and add deterministic fixtures.

7. EXPLAINABILITY

Record evidence, confidence, provenance, and unknown states.

8. RESOURCE CONTROL

Bound runtime, memory, disk, and corpus use.

9. INTERRUPTION RECOVERY

Persist enough state to resume expensive work.

10. END-OF-PHASE AUDIT

Require the builder to distinguish:

* complete
* partial
* experimental
* blocked
* not implemented

11. NO AUTOMATIC ASSUMPTION OF THE NEXT PHASE

Completion of one phase does not authorize the next.

The builder’s report must be reviewed before the following work order is
written.

======================================================================
PROGRAM SUCCESS DEFINITION
==========================

The program succeeds when Cinelingus can demonstrate the following complete
cycle for multiple cinematic properties:

OBSERVE

The system detects a cinematic property using explicit evidence.

REPRESENT

The property exists in a stable, versioned model.

TRANSFORM

A transformation law deliberately changes or transfers it.

RENDER

The altered structure is compiled into a film.

VERIFY

The system measures whether the intended change occurred.

REPAIR

The system attempts bounded correction when the law or delivery contract fails.

The minimum final demonstration should include:

* semantic dialogue transformation
* dialogue-function preservation
* character-aware placement
* performance-role transformation
* measurable progression across a film
* cross-film transfer of a cinematic property
* composition of multiple transformation laws

The final system must remain:

* local-first
* inspectable
* uncertainty-aware
* provenance-preserving
* contract-driven
* reversible where practical
* compatible with existing apparatuses
* protective of source media
